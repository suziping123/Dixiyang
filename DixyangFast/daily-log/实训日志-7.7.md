今天的实训围绕**基于RAG + LangChain Agent的智能法律问答与邮件发送系统**展开。我在之前搭建的FastAPI聊天服务基础上，集成了以下核心能力：

#### （1）RAG检索系统的搭建

**向量数据库**：我使用ChromaDB作为向量数据库，数据集来自`法律数据集.csv`，包含1000条中国法律条文。嵌入模型采用BGE-M3（`/home/lijiajia/models/bge-m3`），将每条法律文本转换为768维向量存入ChromaDB，集合名称为`lawdata`。

核心实现位于`app/chat/ai/CreatBase.py`：

```python
from sentence_transformers import SentenceTransformer
import chromadb

model = SentenceTransformer("BAAI/bge-m3")
client = chromadb.PersistentClient(path="app/chat/chroma_data/")
collection = client.get_or_create_collection(
    name="lawdata",
    embedding_function=model.encode
)
```

**重排序**：为了提升检索质量，我引入了BGE-Reranker（`/home/lijiajia/models/bge-reranker-large`），对ChromaDB初步召回的前20条结果进行深度重排序，最终保留前5条最相关的条文。

```python
from sentence_transformers import CrossEncoder
reranker = CrossEncoder("BAAI/bge-reranker-large")
pairs = [[query, doc] for doc in documents]
scores = reranker.predict(pairs)
```

#### （2）LangChain Agent的构建

通过`create_react_agent`函数创建智能代理，绑定两个核心工具：

- **`legal_rag_search`**：执行法律条文检索，接收用户问题后调用RAG pipeline返回最相关的法律条文

### 三、实验收获

#### RAG技术的深入理解

通过本次实训，我深入理解了RAG（Retrieval-Augmented Generation）的完整工作流程：

- **嵌入模型**：文本 → 高维向量（语义空间映射）
- **向量检索**：余弦相似度查找最接近的文档
- **重排序**：用交叉注意力模型对初步结果进行精细排序，大幅提升准确率
- **Agent感知**：Agent根据用户Query自动判断是否需要检索，将检索结果作为上下文供大模型生成

特别是在法律领域的应用中，RAG的优势非常明显——避免了LLM产生虚构法条（幻觉），确保回答基于真实的法律条文。

#### LangChain Agent的实际运用

`create_react_agent`是LangGraph提供的预置Agent实现，基于ReAct（Reasoning + Acting）范式：

1. **思考（Thought）**：Agent根据当前状态决定下一步行动
2. **行动（Action）**：调用工具（检索、发邮件）
3. **观察（Observation）**：获取工具返回结果
4. **循环**：继续思考 → 行动 → 观察，直到完成任务

我最初遇到的主要问题是**工具参数格式**——Agent无法直接传递多个参数给工具，因此所有工具必须设计为接收单一JSON字符串。通过将`send_email`的三个参数（to、subject、content）封装为一个JSON字符串，成功解决了这个问题。

#### 模型缓存的重要性

BGE-M3嵌入模型约1.2GB，BGE-Reranker约1.6GB，如果每次请求都重新加载，系统完全不可用。通过模块级全局缓存：

```python
_embedding = None  # 全局单例
```

确保了模型只加载一次，后续请求直接复用，响应时间从10+秒降低到2-3秒。

#### SMTP邮件发送

学会了使用QQ邮箱的SMTP服务发送邮件：
- QQ邮箱需要开启SMTP服务并生成授权码（非登录密码）
- SSL加密连接使用端口465
- 邮件正文需要处理中文编码（`utf-8`）

#### 环境变量管理

敏感信息（API密钥、SMTP授权码）通过环境变量注入：
- `DS_API_KEY`：DeepSeek API密钥
- `SMTP_PASSWORD`：QQ邮箱SMTP授权码

放在`.env`文件中，避免硬编码在代码里，提升安全性。

### 四、实验感悟

**最让我印象深刻的是Agent的自主决策能力**。当我输入"有关恐怖主义的中国法律条文发到邮箱"时，Agent自动完成了两个步骤：先检索法律条文，再发送邮件。整个过程中，我只需要提出需求，Agent自主规划并执行，这就是AI Agent的魅力所在。

**RAG在实际应用中的价值非常大**。在没有RAG的情况下，LLM只能依靠训练数据中的记忆来回答法律问题，容易产生"幻觉"——编造不存在的法条。有了RAG，每次回答都有真实的法律条文作为依据，结果可信度高了很多。

**重排序模型的引入是关键一步**。最初只使用BGE-M3进行向量检索时，返回的Top-5结果中有一些相关性不高的条文。加入BGE-Reranker后，检索准确率显著提升，最相关的条文总能排在最前面。

**关于调试Agent**，我也积累了一些经验。Agent的思考过程比较复杂，很难一次性调好。我通过在提示词中明确指定工具调用顺序和参数格式，大大提高了Agent的成功率。同时，观察Agent的中间输出（thought/action/observation）对于定位问题非常有帮助。

**最后，系统的健壮性还需要加强**。目前邮件发送是同步阻塞的，如果SMTP服务器响应慢，整个请求都会被阻塞。后续可以改为异步任务队列（如Celery），将邮件发送放到后台执行。

### 五、课程 / 实验建议

1. **异步任务**：邮件发送等耗时操作建议使用消息队列（Redis + Celery）异步处理，避免阻塞API响应。

2. **检索增强**：当前只使用了ChromaDB，后续可以对比Elasticsearch全文检索 + 向量检索的混合方案，进一步提升召回率。

3. **用户认证**：如果系统需要多用户使用，应添加用户认证机制，确保邮件发送操作的安全性。

4. **前端可视化**：可以开发前端界面，让用户直接在浏览器中输入问题、查看检索结果、确认后再发送邮件。

5. **更多工具**：Agent目前只有两个工具（检索+邮件），可以扩展更多工具，如PDF生成、短信通知、知识库管理等。

6. **日志监控**：建议添加请求日志和错误监控，便于排查问题和统计使用情况。
